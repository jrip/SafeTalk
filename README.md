# SafeTalk

SafeTalk - сервис для определения токсичности текста в диалогах.

Внешние системы передают диалоги через API или web-интерфейс и получают результат анализа токсичности.

## Зачем это нужно

В чат-системах (например, в клиентской поддержке) важно вовремя обнаруживать конфликтные ситуации.  
SafeTalk решает две ключевые задачи:

- оперативно сигнализирует о потенциальном конфликте, чтобы старший сотрудник мог подключиться к диалогу;
- позволяет анализировать переписки задним числом и находить проблемные разговоры.


## Архитектура SafeTalk

SafeTalk представляет собой RESTful API и состоит из независимых модулей.
Для реализации личного кабинета пользователя и демонстраций функционала будет отдельное приложение на react/typeScript, используещее это API

Каждый модуль включает:
- HTTP-контроллер (если модулю нужен внешний API-слой);
- сервис или набор сервисов с бизнес-логикой;
- внутренний интерфейс сервиса (для использования внутри модуля);
- внешний интерфейс сервиса (для использования другими модулями);
- модели для хранения в БД и, при необходимости, модели для работы с очередями.

## Модули сервиса

Сервис состоит из пяти модулей: `Users`, `Neural`, `Billing`, `History`, `Feedback`.
`Neural` использует `Users`, `Billing` и `History`.  
`Feedback` использует `History`.

---

### Users

Модуль управления пользователями.

**Внешние API-методы:**
- регистрация;
- получение токена авторизации;
- получение информации о текущем пользователе;
- обновление информации о текущем пользователе.

**Модель `User` (API):**
- `id: uuid` - первичный ключ, случайный UID;
- `email: string` - логин (email), пользователь не может изменить;
- `password: string` - пароль в хешированном виде с солью, пользователь может изменить;
- `name: string` - имя пользователя, пользователь может изменить;
- `role: string` - роль пользователя, по умолчанию `user`.

**Внутренний сервис `users`:**
- используется внутри системы.

**Модель `User` (БД):**
- `id: uuid` - первичный ключ, случайный UID;
- `email: string` - логин (email);
- `password: string` - пароль в хешированном виде с солью;
- `name: string` - имя пользователя;
- `role: string` - роль пользователя, по умолчанию `user`;
- `allow_negative_balance: bool` - если `true`, баланс может уходить в минус;
- `ctime: datetime` - дата и время регистрации.

### Neural

Модуль, реализующий основную задачу: определение уровня токсичности диалога.

**Внешние API-методы:**
- получить токсичность диалога.
- создать задачу на определение токсичности пачки диалогов
- получить результат задачи по пачке диалогов

**Внутренние API-методы:**
- получить токсичность диалога.
- создать задачу на определение токсичности пачки диалогов
- получить результат задачи по пачке диалогов

**Модели (текущий черновой вариант):**
- `Dialog`:
  - `text: string` - текст диалога;
- `ToxyOutput`:
  - `isToxic: boolean` - флаг токсичности.

> Примечание: финальный состав полей может измениться после выбора конкретного ML-механизма.

---

### Billing

Модуль учета расхода токенов.

**Внешние API-методы:**
- получить количество доступных токенов.

**Модель `Balance` (API):**
- `tokenCount: int` - количество токенов.

**Внутренние API-методы (публичные для других модулей):**
- `spendTokens(count: int)` - списать токены;
- `getCountTokens() -> int` - получить текущий баланс.

**Модель `Tokens` (БД):**
- `id: uuid` - случайный UUID;
- `user_id: uuid` - идентификатор пользователя;
- `ctime: datetime` - время создания записи;
- `count: int` - количество списанных токенов (отрицательное значение) или начисленных токенов (неотрицательное значение).

---

### History

Модуль хранения истории запросов к API.

**Внешние API-методы:**
- получить историю запросов к API.

**Модель `History` (API):**
- `id: uuid` - первичный ключ;
- `ctime: datetime` - время создания записи;
- `request: string` - диалог, отправленный на анализ;
- `result: string` - текстовое представление результата анализа;
- `user_id: uuid` - идентификатор пользователя.

**Внутренние сервисные методы:**
- получить историю API-запросов по пользователю (приватный для модуля);
- сохранить запрос к API (публичный для других модулей).

**Модель `History` (БД):**
- `id: uuid` - первичный ключ;
- `ctime: datetime` - время создания записи;
- `request: string` - диалог, отправленный на анализ;
- `result: string` - текстовое представление результата анализа;
- `user_id: uuid` - идентификатор пользователя.

---

### Feedback

Модуль сбора обратной связи по результатам анализа (для последующего дообучения).

**Внешние API-методы:**
- отправить фидбек на анализ.

**Модель `Feedback` (API):**
- `id: uuid` - `history_id` записи из истории;
- `isToxic: boolean` - ожидаемая токсичность (как должно было быть).

**Внутренние сервисные методы:**
- сохранить фидбек на анализ (приватный для модуля).

**Модель `Feedback` (БД):**
- `id: uuid` - `history_id` записи из истории;
- `isToxic: boolean` - ожидаемая токсичность (как должно было быть).
