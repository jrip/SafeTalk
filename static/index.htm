Токсичность текста — без очереди у вашей команды

SafeTalk — сервис проверки сообщений и диалогов на токсичность: каталог моделей и правила агрегации результатов. Запросы принимает REST API, тяжёлый ML уходит в RabbitMQ и обрабатывается несколькими воркерами — вы не упираетесь в один процесс и можете масштабировать поток.

Кто ваша аудитория — и что им предложить

Один продукт, разные «покупатели» внутри компании. Ниже — типичные роли и как SafeTalk закрывает их задачу без лишней суеты.

Крупные клиенты

Маркетплейсы, банки, телеком, EdTech

Высокий поток пользовательского текста: отзывы, чаты поддержки, комментарии, UGC.

  • Предлагаем: API + очередь — пики нагрузки не валят API-процесс; воркеры масштабируются горизонтально; аудит и биллинг по факту объёма.
  • Дообучение: адаптация модели под ваш домен (лексика поддержки, внутренний сленг, отраслевые формулировки) — отдельная дорожка с данными и политикой качества.
  • Специальные условия: объёмы, сроки, пилоты и коммерческие рамки под контракт — не «как с сайта», а под ваш закупочный процесс.
  • Кастомные интеграции: VPC / выделенные воркеры, нестандартные потоки (очереди, webhooks, выгрузки в ваш SIEM / DWH), совместная схема с ИБ и комплаенсом.

Средний и малый бизнес

Сервисы с чатом, CRM, небольшие площадки

Нужно «поймать» оскорбления и угрозы до эскалации, не нанимая армию модераторов на первую линию.

  • Предлагаем: простой контракт API — создать задачу, опросить статус, получить is_toxic и разбор меток.
  • Ценность: несколько моделей в каталоге — выбираете вариант под задачу и бюджет (разная цена за символ, разный компромисс скорость/качество).
  • Внедрение: несколько вызовов в день из вашего бэкенда или скрипта отдела качества.

Что именно мы предлагаем

Коротко: не «чёрный ящик в чате», а предсказуемый сервис с очередью, учётом и прозрачным ответом по задаче.

Архитектура под нагрузку

Publisher (FastAPI) принимает POST /predict, пишет задачу в БД и отправляет JSON в одну durable-очередь RabbitMQ. Consumers (ML-воркеры) забирают сообщения и дописывают результат.

Разные модели — одна точка входа

Каталог активных моделей: переключайтесь под сценарий (жёстче/мягче, дороже/дешевле). Ответ по задаче — вероятности, агрегированное решение и текстовое резюме для ваших правил.

Экономика токенов

Списание привязано к длине текста и тарифу модели. Для крупного клиента — прогнозируемый чек; для малого — нет платы «за простой сервер».

Модели под ваши сценарии

По договорённости возможно дообучение (fine-tuning) или иная адаптация моделей под ваш домен: лексика поддержки, отраслевой жаргон, специфика площадки. Отдельно согласовываются данные для обучения, метрики качества, сроки и правовые ограничения — чтобы сигнал ML совпадал с вашими правилами модерации.

Как это работает для интегратора

Поток, который можно показать архитектору и бизнесу — без лишней магии.

  1. Авторизация — POST /auth/login, токен в заголовке для защищённых ручек.
  2. Создание задачи — POST /predict с model_id и текстом; в ответе только task_id (асинхронный контракт).
  3. Очередь — сообщение с task_id, признаками и временем уходит в RabbitMQ; свободный воркер забирает работу (round-robin между воркерами).
  4. Результат — клиент опрашивает GET /predict/{task_id} до статуса completed или обрабатывает ошибку/таймаут по своей политике.

Примеры использования

Готовые истории «зачем это в проде» — можно переносить в презентацию или ТЗ на интеграцию.

Премодерация отзывов

Перед публикацией на витрине — асинхронная проверка; при подозрении — в ручную модерацию, иначе — в эфир.

Очередь обращений в поддержку

Автоматическая метка «возможная агрессия» для супервизора и приоритизации ответа.

Внутренние переписки (HR / комплаенс)

Выборочный скрининг при соблюдении локальных политик и согласий — технически тот же API.

Бот в Telegram

После настройки webhook — единый пайплайн: сообщения пользователей попадают в ваш обработчик и при необходимости в ML-очередь.

Важно: SafeTalk даёт технический сигнал токсичности, а решения «банить / не банить» и юридические рамки остаются на стороне продукта и комплаенса.

Для разработчиков и продакта

Разработчикам: интеграция на обычном REST — вход по логину и паролю, дальше токен в заголовке; каталог моделей, создание задачи на проверку текста и опрос статуса до завершения. Ответ на создание задачи приходит быстро: вы получаете task_id и не держите HTTP-соединение на время работы ML — обработка идёт в фоне через очередь, что упрощает масштабирование и переживание пиков трафика.

Продакту и владельцу продукта: в открытой документации API видно, какие поля возвращаются по задаче (статус, списание, итог токсичности и разбор по классам) — это можно сразу закладывать в ТЗ, сценарии модерации и оценку сроков внедрения, без «чёрного ящика» на стороне интеграции.
