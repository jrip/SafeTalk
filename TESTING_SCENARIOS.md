# Сценарии тестирования

Ниже приведены пользовательские сценарии для ручного тестирования системы. К каждому сценарию можно приложить скриншот интерфейса как подтверждение выполнения.

## 1. Регистрация пользователя

**Шаги:**
- Открыть форму регистрации.
- Ввести имя, email и пароль.
- Отправить форму.

**Ожидаемый результат:**
- Новый пользователь успешно создается в системе.

**Что проверяет:**
- Система позволяет создать нового пользователя и начать дальнейшую работу с сервисом.

## 2. Подтверждение email

**Шаги:**
- Получить код подтверждения.
- Ввести код в форме подтверждения.

**Ожидаемый результат:**
- Email успешно подтвержден.

**Что проверяет:**
- Пользователь должен подтвердить email перед полноценным использованием системы.

## 3. Авторизация пользователя

**Шаги:**
- Открыть форму входа.
- Ввести корректные учетные данные.
- Выполнить вход.

**Ожидаемый результат:**
- Пользователь успешно входит в систему.

**Что проверяет:**
- Только зарегистрированный и подтвержденный пользователь получает доступ к защищенным функциям.

## 4. Повторная авторизация

**Шаги:**
- Выйти из аккаунта.
- Повторно выполнить вход с теми же данными.

**Ожидаемый результат:**
- Повторная авторизация проходит успешно.

**Что проверяет:**
- Механизм входа работает стабильно не только при первом использовании, но и при повторном входе.

## 5. Ошибка при неверных данных авторизации

**Шаги:**
- Ввести неправильный пароль или некорректный логин.
- Попробовать выполнить вход.

**Ожидаемый результат:**
- Система показывает сообщение об ошибке и не выполняет вход.

**Что проверяет:**
- Система не должна пускать пользователя при неверных учетных данных.

## 6. Просмотр текущего баланса

**Шаги:**
- Авторизоваться.
- Открыть раздел с балансом пользователя.

**Ожидаемый результат:**
- На экране отображается текущий баланс.

**Что проверяет:**
- У пользователя есть доступ к информации о количестве доступных кредитов.

## 7. Пополнение баланса

**Шаги:**
- Выполнить пополнение баланса.
- Обновить страницу или повторно открыть раздел баланса.

**Ожидаемый результат:**
- Баланс увеличивается на указанную сумму.

**Что проверяет:**
- Система корректно сохраняет и отображает изменение баланса после пополнения.

## 8. Успешная отправка ML-запроса

**Шаги:**
- Открыть форму анализа текста.
- Ввести корректный текст.
- Запустить обработку.

**Ожидаемый результат:**
- Запрос успешно отправляется на обработку.

**Что проверяет:**
- Пользователь может воспользоваться основной функцией системы.

## 9. Получение результата ML-запроса

**Шаги:**
- Дождаться завершения обработки отправленного текста.
- Просмотреть результат анализа.

**Ожидаемый результат:**
- Пользователь получает результат предсказания.

**Что проверяет:**
- Система не только принимает запрос, но и возвращает корректный результат пользователю.

## 10. Корректное списание кредитов после ML-запроса

**Шаги:**
- Зафиксировать баланс до выполнения ML-запроса.
- Выполнить успешный ML-запрос.
- Проверить баланс после выполнения.

**Ожидаемый результат:**
- После успешного ML-запроса баланс уменьшается корректно.

**Что проверяет:**
- ML-запрос является платной операцией, и система должна правильно списывать кредиты только после успешного выполнения.

## 11. Обработка некорректных входных данных

**Шаги:**
- Ввести пустой текст или некорректные данные в форму анализа.
- Отправить запрос.

**Ожидаемый результат:**
- Система показывает сообщение об ошибке и не выполняет обработку.

**Что проверяет:**
- Система должна валидировать входные данные и не запускать обработку некорректного запроса.

## 12. Запрет выполнения ML-запроса при недостаточном балансе

**Шаги:**
- Убедиться, что на балансе недостаточно средств.
- Попробовать отправить ML-запрос.

**Ожидаемый результат:**
- Запрос не выполняется, система сообщает о недостаточном балансе.

**Что проверяет:**
- Пользователь не может использовать платную функцию при отсутствии достаточного количества кредитов.

## 13. История транзакций

**Шаги:**
- Выполнить пополнение и списание.
- Открыть раздел истории операций.

**Ожидаемый результат:**
- В истории отображаются операции пополнения и списания.

**Что проверяет:**
- Все финансовые действия пользователя должны фиксироваться в системе.

## 14. История ML-запросов

**Шаги:**
- Выполнить один или несколько ML-запросов.
- Открыть раздел истории запросов.

**Ожидаемый результат:**
- В истории отображаются выполненные ML-запросы и результаты обработки.

**Что проверяет:**
- Система сохраняет историю использования основной функции сервиса.

## 15. Отсутствие списания при ошибке ML-запроса

**Шаги:**
- Зафиксировать баланс до отправки некорректного ML-запроса.
- Отправить некорректный запрос.
- Проверить баланс после ошибки.

**Ожидаемый результат:**
- При ошибке ML-запроса баланс не изменяется.

**Что проверяет:**
- Списание кредитов должно происходить только за успешную операцию, а не за неудачную попытку.

## 16. Обработка частично валидных данных при пакетной загрузке

**Шаги:**
- Открыть форму пакетной загрузки данных.
- Загрузить набор, в котором часть записей корректная, а часть превышает допустимую длину.
- Запустить предварительную проверку или обработку.

**Ожидаемый результат:**
- Система корректно выявляет записи, нарушающие ограничение по длине, и показывает ошибку валидации.
- Корректные данные обрабатываются в соответствии с реализованной логикой интерфейса и сервиса.

**Что проверяет:**
- Система корректно работает со смешанными данными и проверяет ограничения до запуска ML-обработки.

## Вывод по результатам тестирования

В ходе тестирования были проверены основные пользовательские сценарии системы: регистрация и авторизация пользователя, работа с балансом, отправка ML-запросов, списание кредитов, просмотр истории операций и валидация данных при пакетной загрузке.

По результатам проверки система корректно выполняет ключевые бизнес-сценарии: пользователь может зарегистрироваться и войти в систему, баланс отображается и обновляется после пополнения, ML-запросы успешно обрабатываются, а при их выполнении производится корректное списание кредитов.

Также была проверена обработка ошибочных ситуаций: система не допускает авторизацию при неверных данных, отклоняет некорректные ML-запросы, корректно обрабатывает ограничения по длине при пакетной загрузке и не выполняет обработку при недостаточном балансе. При возникновении ошибки списание средств не производится.

История транзакций и история ML-запросов сохраняются и позволяют сопоставить действия пользователя с результатами работы системы.

Таким образом, система в целом функционирует корректно и соответствует основным требованиям, предъявляемым в рамках практического задания по тестированию работоспособности.
